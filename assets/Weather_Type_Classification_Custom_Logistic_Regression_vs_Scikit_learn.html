1. Preparation and Overview¶

Classification Task¶

This dataset is meant for machine learners. It explicitly states that the data is made up for learning, so this model wouldn't be useful in deployment. That being said the task of identifying weather conditions is old as time and something I imagine ML models are already being trained to help with (if not already deployed in some places).

The target variable we are predicting in this dataset is the "Weather Type", Rainy, Sunny, Cloudy, or Snowy. This task is one done by weather apps already, giving people a rough estimate of what the weather is like based on many factors so they can prepare to go about their day. I believe a model like this could be deployed and fed data periodically throughout the day, at which point it can output what the weather is like in places for users.

This task is not one which requires an extraordinarily high amount of accuracy, as getting the weather wrong is not the biggest of deals. At worst a user of the model may be dressed innapropriately for the weather. In most cases this may lead to mild discomfort, or a loss of time due to changing. But a classifier of this type cannot be too innaccurate or it will be discarded. People remember when their weather app gave them the wrong information. If it is wrong to often, people will abandon it in favor of something else. It must be rememebered that "Window Weather" is an option for most people, so a classifier needs to have decently high accuracy to incentivize people over that.

def train_test_split(X, y, test_size=0.2, random_state=None):
  if random_state:
    np.random.seed(random_state)

  # Create an array of indices and shuffle them
  indices = np.arange(X.shape[0])
  np.random.shuffle(indices)

  # Calculate the split index
  split_idx = int(X.shape[0] * (1 - test_size))

  # Split the indices into training and testing sets
  train_indices = indices[:split_idx]
  test_indices = indices[split_idx:]

  X_train, X_test = X[train_indices], X[test_indices]
  y_train, y_test = y.iloc[train_indices], y.iloc[test_indices]

  return X_train, X_test, y_train, y_test

In researching the amount of data needed for training models, I came across the "10 times Rule", defined as "having at least 10 examples for each feature or predictor variable in your model". This rule is not perfect especially considering the variety of models and predicting tasks out there, many of which require much more data or have many many features and therefore realistically can't have 10 times that amount of data. As This rule was developed in the time of primarily linear and logistic regression models, specifically the type of model we are working with, it would seem too be appropriate for evaluating the appropriateness of our data here.

We have 10 features, so 10x10 would be 1000 instances for a hopefully appropriate dataset. This dataset has 13,000+ instances, which is more than adequate given the 10 times rule.

Moreover, (considering the dataset is one made for learning), all the target classes have an equal number of instances of 3,300. This very equal distribution lessens the fear that removing to much of one group might have negative effects as it is very unlikely.

So there is no reason to lesson the amount of testing data to 10%. 80%-20% is completely fine to use in this case, although 70%-30% may also be appropriate considering the amount of data we have.

2. Modeling¶

Creating a Custom Logistic Regression Model¶

Here, we implemented a custom, one-versus-all logistic regression classifier using numpy. The implementation follows object-oriented conventions like scikit-learn. Addtitionally, the the logistic regression classifier has the following functionality added:

  • Ability to choose optimization technique when class is instantiated: either steepest ascent (gradient descent), stochastic gradient ascent, and {Newton’s method/Quasi Newton methods}.

One-versus-all (OvA) is a strategy for multi-class classification where a binary classifier is trained for each class against all others. For n classes, n binary classifiers are trained. This approach transforms an n-class problem into n binary classification problems, where each problem distinguishes a given class from all other classes combined.*

# Implementation tips:
# To implement these methods, you would need to define functions for the sigmoid,
# loss, gradient, and (for Newton’s method) Hessian calculations.
# Then, create update rules based on each method.

class LogisticRegressionOVA:

  def __init__(self, solver, learning_rate, num_iterations, penalty, C):
    self.solver = solver
    self.learning_rate = learning_rate
    self.num_iterations = num_iterations
    self.models = []  # List to hold models for each class
    self.penalty = penalty  # Regularization type: 'none', 'l1', 'l2', 'elasticnet'
    self.C = C  # Regularization strength (cost)


  def sigmoid(self, z):
    return 1 / (1 + np.exp(-z))

  def loss(self, y, y_pred):
    m = len(y)
    loss = -(1 / m) * np.sum(y * np.log(y_pred) + (1 - y) * np.log(1 - y_pred))
    return loss

  def compute_gradients(self, X, y, y_pred, theta):
    m = len(y)
    y = y.to_numpy().reshape(-1, 1)  # Ensure y is a column vector
    gradient = (1 / m) * np.dot(X.T, (y_pred - y))

    # Add regularization term to the gradient
    if self.penalty == 'l1':
        gradient[1:] += (self.C / m) * np.sign(theta[1:])
    elif self.penalty == 'l2':
        gradient[1:] += (self.C / m) * theta[1:]
    elif self.penalty == 'elasticnet':
        gradient[1:] += (self.C / m) * (theta[1:] + np.sign(theta[1:]))


    return gradient

  def fit(self, X, y):
    m, n_features = X.shape
    num_classes = len(np.unique(y))  # Number of unique classes

    # Add bias term to X
    X = np.c_[np.ones((m, 1)), X]  # Add column of ones to X for the bias term

    self.models = []  # initialize self.models as an empty list

    for i in range(num_classes):
      # Create binary labels for class i
      y_binary = (y == i).astype(int)

      # Initialize theta for the current class
      theta = np.zeros((n_features + 1, 1))  # +1 for bias term

      # Fit the model for the current class
      for _ in range(self.num_iterations):
        z = np.dot(X, theta)
        y_pred = self.sigmoid(z)
        loss = self.loss(y_binary, y_pred)
        gradient = self.compute_gradients(X, y_binary, y_pred, theta)

        # Gradient Descent
        if self.solver == 'steepest_ascent':
          theta = theta - self.learning_rate * gradient

        # Stochastic gradient ascent updates
        elif self.solver == 'stochastic':
          theta = self.stochastic_gradient_ascent(X, y_binary, theta)

        # Newton's method updates
        elif self.solver == 'newton':
          theta = self.newton_method(X, y_binary, theta)

        else:
          raise ValueError("Invalid solver specified")

      # Store the model (theta) for class i
      self.models.append(theta)


  def stochastic_gradient_ascent(self, X, y, theta):
    # Stochastic gradient descent update
    i = np.random.randint(len(y))
    X_i = X[i:i+1]  # Select the i-th sample (row)
    y_i = y[i:i+1]  # Select the corresponding label

    h = self.sigmoid(np.dot(X_i, theta))
    gradient = self.compute_gradients(X_i, y_i, h, theta)
    theta = theta - self.learning_rate * gradient
    return theta

  def newton_method(self, X, y, theta):
    # Newton's method update (simplified)
    h = self.sigmoid(np.dot(X, theta))
    hessian = np.dot(X.T, X * h * (1 - h)) / len(y)
    gradient = self.compute_gradients(X, y, h, theta)
    theta = theta - np.dot(np.linalg.pinv(hessian), gradient)
    return theta


  def predict(self, X):
    # Add bias term to X
    X = np.c_[np.ones((X.shape[0], 1)), X]

    # Compute the probabilities for each class
    num_samples = X.shape[0]  # Get the number of samples in X
    probs = np.zeros((num_samples, len(self.models)))  # Initialize probs with correct shape

    for i, model in enumerate(self.models):
      z = np.dot(X, model)  # Calculate z for all samples
      probs[:, i] = self.sigmoid(z).ravel()  # Store probabilities for class i

    # Predict the class with the highest probability
    predictions = np.argmax(probs, axis=1)  # Choose the class with the highest probability

    # Reshape to a column vector with the correct number of rows
    return predictions.reshape(-1, 1)

I do not believe our project introduces any "data snooping". This is a bit hard to describe as in my research, data snooping is a bit more of an unethical practice than a small mistake you make while putting a model together. In many of the cases I read, data snooping referred to dishonest practices of running tests/adjusting things until you got a good result, and not considering the possible insignifianct reasons (for example, data which shows a false coorelation) data might appear signficiant. In our case, we used all of the parameters, so there was no searching through the data to introduce bias. We randomized the data which got split into training and testing, so there was no bias there. We also aren't testing on the same data we trained on, so there's no problems there either. We also got decent results from our model, so there was no going back and messing around with things to try and make it artificially better.

We can check for overfitting when we get our testing results. If we tested using the training data and got extremely high accuracy results, but then used data from outside the training and got poor results, it would be a sign the data is overfit. We could also get the equation after the training and graph it next to the data to see if it's bounded to closely to the data. Considering we have an accuracy of the mid 80's while testing on data that was not used during training, I do not believe our model is overfit.

Comparison¶

Now, let's compare the results from both of our models...

  • Confusion Matrix (each of our four classes has all four of the following)
    • True Positives: The elements where both the predicted class and the observed class match.
    • False Positives: The elements where we are predicting one class, but it is observed to not be that class.
    • False Negatives: The elements where we are not predicting one class, but they are observed to be in that class anyway.
    • True Negatives: The elements that we are both not predicting and also do not belong to our predicted class

Our Model:

Scikit Learns Model:

Based solely on the confusion matrix of the two, the Scikit learn model is slightly better than our model. In all of the incorrect squares (except Sunny misclassified as Snowy) it is lower or the same as our model, and all of the true positive squares are higher by between 1 and 33. While it is not excelling by leaps and bounds in all of them, it is consistently a little better across the board.

  • Precision
    • Precision is the percentage of observed positives (for one class) out of all predicted positives (for that same class). We can have one precision with a weighted average, where each class's precision is weighted based on how many samples belong to that class in the dataset, giving more importance to the performance on larger classes and less importance to smaller classes

3. Deployment¶

After reviewing all the metrics side by side, while we created a model with decent metrics, I have to recommend Scikit Learns model over ours. In all metrics it preformed better than our own, so it is clearly the superior model, even if only by a little. And, when it comes to the weather, that little bit could mean a lot. As discussed at the beginning, while providing an incorrect assessment of the weather generally will not harm anyone, if it is wrong to often people will lose faith in the application. Therefore, the slightly higher performance, could mean quite a bit in deployment.
